# 工作流模式

Stitch 前端工作流有四种模式。原则是：选能完成任务的最轻模式，不要默认把每次任务都跑成完整流程。

| 模式 | 适合场景 | 是否使用 Stitch | 输出 |
|---|---|---:|---|
| 极简模式 | 产品想法还在成形，第一次 UI 方向没定 | 推荐官网，MCP 可选 | 产品简报 + Stitch 提示词 |
| 快速模式 | 普通前端任务，速度更重要 | 不使用 | 直接实现 + 基础验证 |
| 设计模式 | 视觉方向很重要 | 调用一次 MCP | Stitch 参考 + 工程实现 |
| 完整模式 | 重要 demo、设计系统、仓库示例 | 使用 | 预览 + Stitch 资产 + 实现 + 多视口验证 |

## 极简模式

当用户第一次设计某个产品时，优先使用极简模式。这个阶段最重要的通常不是立刻写代码，而是先把产品感觉和 UI 方向摸出来。

推荐流程：

1. 打开 Stitch：<https://stitch.withgoogle.com>
2. 粘贴 Codex 生成的产品简报或 Stitch 提示词。
3. 自己在 Stitch 里探索几版 UI 方向。
4. 把选中的截图、导出的 HTML、DESIGN.md 或项目地址交给 Codex。
5. 让 Codex 把选定方向落到真实项目里。

极简模式成本最低，也能避免产品还没想清楚时就过早工程化。

## 快速模式

当用户已经知道要什么页面、不需要 AI 做视觉探索时，使用快速模式。

Codex 应该：

- 检查项目结构
- 直接实现页面或组件
- 做语法检查和一两个关键视口的视觉验证
- 除非用户明确要求，否则不调用 Stitch

## 设计模式

当用户希望 AI 帮忙给出视觉方向，但不需要完整资产流水线时，使用设计模式。

Codex 应该：

- 写一份简短页面简报
- 调用一次 Stitch
- 保存截图或元数据
- 把视觉方向重写为可维护的项目代码
- 验证桌面和移动端

## 完整模式

只有当工作流本身需要被记录，或者页面足够重要、值得花更多时间时，才使用完整模式。

Codex 应该：

- 创建本地 HTML 预览
- 调用 Stitch
- 保存原始 JSON、截图、HTML 和设计上下文
- 重写为项目代码
- 验证桌面、移动端和窄屏移动端

这个模式故意更重，适合证明流程、做教程或做重要页面，不应该成为默认模式。
